# Hermes Desktop Page Requirements Reset

**Date:** 2026-04-14  
**Scope:** Desktop shell, onboarding, `对话 / 渠道 / 技能 / 任务 / 设置`  
**Status:** 页面级产品需求重置版  
**Goal:** 把 Hermes Desktop 从“信息很多但价值不高的功能台”重置成“以对话为核心、配置门槛低、像桌面客户端的产品界面”。

---

## 1. 产品结论

Hermes Desktop 不是工作台，不是控制中心，也不是后台管理系统。

它首先是一个桌面端 AI 客户端，用户打开它的核心目的只有三类：

1. 直接开始或继续一段对话
2. 把渠道、技能、任务这些辅助能力接起来
3. 在必要时完成模型与偏好设置

因此，新的页面需求必须建立在下面这条产品判断上：

> 默认界面只展示“当前任务真正需要的信息”，其余能力通过逐层进入的方式暴露，而不是一次性平铺在前端。

这份文档是新的页面级需求基线。此前偏 `dashboard / workbench / control center` 的表达方式，不再作为视觉或交互参考。

---

## 2. 当前问题

目前页面方向主要有四个问题：

### 2.1 页面像管理台，不像桌面客户端

- 壳子太重
- 卡片过多
- 摘要区过多
- 页面在解释自己，而不是帮助用户完成任务

### 2.2 信息价值排序错误

很多页面默认展示了技术上“存在”的信息，但这些信息不一定是用户真的关心的，例如：

- 路径
- session id
- 文件名
- 大量原始状态字段
- 内部配置树
- 解释性说明文案

这些信息并非完全没价值，但默认展示会打断注意力。

### 2.3 页面职责边界不清

聊天页混入了过多系统信息；设置页试图承担知识中心和运行态总览的职责；渠道、技能、任务页也有把信息全堆出来的倾向。

### 2.4 设置页建模过度

模型设置此前引入了超前的“任务路由”表达，这和 Hermes 当前真实能力并不一致。普通用户真正需要的是：

> 我默认用哪个模型？这个接口通不通？保存以后能不能直接开始用？

---

## 3. 产品原则

## 3.1 一个页面只服务一个主任务

- `对话` 页只服务对话
- `渠道` 页只服务接入与可用性
- `技能` 页只服务能力开关与安装
- `任务` 页只服务调度与执行
- `设置` 页只服务偏好与配置

任何页面都不应在默认状态下承担“总览全系统”的职责。

## 3.2 健康状态应该安静

当 Hermes 正常可用时，页面不需要反复强调：

- 版本号
- 路径
- 运行态解释
- 环境细节

只有在异常、缺失、未连接时，才需要明显但克制地提醒用户。

## 3.3 简单配置优先，高级能力后露出

默认流程只解决“尽快开始使用”。

高级能力要保留，但必须通过：

- 折叠
- 次级分段
- 详情页
- 高级模式

逐步进入，而不是默认全开。

## 3.4 当前对象与操作必须同屏

桌面客户端偏好设置的关键不是炫技，而是：

- 你点的是谁
- 你改的是谁
- 保存作用于谁

这三件事必须在同一视线范围内完成，避免“上面点概念，下面找表单”。

## 3.5 只展示真实能力，不发明概念

前端表达必须和 Hermes 当前真实能力对齐，不再把不存在的一等能力包装成 UI 主体。

特别是：

- 不再把“任务路由”当作当前 Hermes 已有实体
- 不再发明复杂模型矩阵来制造高级感

## 3.6 文案短、直接、可执行

不要再出现这类文案：

- 当前重点
- 默认投递链路
- 低价值信息收起
- 先恢复 Gateway 再谈调度
- 这里展示的是……

页面不是产品说明书。默认文案只承担三件事：

- 标识对象
- 说明状态
- 提供动作

---

## 4. 全局信息架构

## 4.1 顶层导航

保留五个一级页面：

- 对话
- 渠道
- 技能
- 任务
- 设置

这是当前 Hermes Desktop 的合理顶层结构，不需要再增加“总览”“工作台”“控制台”等入口。

## 4.2 全局壳子

推荐使用轻量桌面壳子：

- 左侧固定导航
- 右侧单一内容工作区
- 顶部只保留当前页面标题与少量上下文动作

壳子的目标是稳定与安静，而不是表达复杂度。

### 壳子要求

- 默认亮色
- 页面层级清楚，但不要重卡片套重卡片
- 阴影轻
- 边框轻
- 大面积留白
- 不要出现海报式头图、概览条、舞台条

## 4.3 页面信息层级

每个页面的信息都按三层处理：

### A 层：默认可见

用户当前任务一定会用到的信息。

### B 层：上下文可见

用户执行某个动作时才需要的信息。

### C 层：高级或排障可见

只在调试、异常或高级配置时需要。

产品设计时，必须先判断信息属于哪一层，再决定是否展示。

---

## 5. 页面级需求

## 5.1 Onboarding / 启动引导

启动引导的唯一目标是把用户带到“可以正常聊天”的状态。

### 主任务

- 检查 Hermes 是否已安装
- 检查是否能读取真实用户环境
- 完成最小模型配置
- 可选连接一个常用渠道

### 页面要求

- 一次只做一件事
- 不暴露完整设置树
- 不要求用户理解 Hermes 内部结构
- 安装检测与环境检测必须基于真实本机环境，而不是假设 Web 沙箱里的路径视图

### 默认流程

1. 检测安装状态
2. 如未安装，给出明确安装动作
3. 如已安装，进入主模型配置
4. 保存成功后直接进入聊天页

### 非目标

- 不在引导页展示完整运行态
- 不在引导页展示大量配置细节
- 不把引导页做成技术诊断面板

---

## 5.2 对话页

对话页是产品核心，默认注意力必须被完整留给当前会话。

### 主任务

- 找到最近会话
- 继续聊天
- 开新会话
- 理解当前回复过程中的关键中间状态

### 布局

推荐使用二级主结构：

- 左侧：会话列表
- 中间：对话流与输入框

右侧详情区只在以下情况出现：

- 用户主动打开详情
- 当前消息出现错误，需要查看细节

### 会话列表默认展示

会话列表项默认只展示：

- 会话标题

必要时可在 hover 或选中态展示极少量补充信息，但不要默认展示：

- 摘要
- 更新时间
- 技术标签堆叠
- 文件名
- session id

### 对话主体默认展示

- 当前会话标题
- 消息流
- 输入框
- 发送状态

当前上下文、模型、成本这类信息可以存在，但必须融合到输入区附近的一行轻量状态中，而不是做成独立信息卡。

### 消息渲染要求

对话流必须原生支持：

- Markdown
- 代码块
- 表格
- 列表
- task list

还必须针对 Hermes 的过程型内容做专门渲染：

- `thinking`
- tool use
- 子代理
- 后台任务
- 任务规划
- 任务列表

### 过程型内容的表现原则

- 作为对话流的一部分出现
- 样式轻量
- 优先一行摘要 + 可展开详情
- 不做重型诊断卡片
- 不做大头像
- 不抢主回复正文的注意力

### 对话页绝对不要默认展示

- Hermes 路径
- 可执行文件位置
- session 文件名
- 原始 JSON 片段
- 工程化解释文案
- 大段运行态说明

### 异常状态

异常应该靠近输入框或当前消息出现，并提供明确动作：

- 重试
- 查看详情
- 去设置

而不是把整个页面切成告警面板。

---

## 5.3 渠道页

渠道页的目标不是展示配置项，而是让用户知道：

> 哪些渠道已经可用，我要怎么接入下一个渠道。

### 主任务

- 查看渠道是否已配置
- 完成渠道接入
- 确认是否可正常投递
- 管理默认投递对象

### 布局

建议使用对象列表 + 当前对象详情的结构。

左侧或中侧列出渠道，右侧展示当前选中渠道的接入方式与状态。

### 渠道列表默认展示

每个渠道卡片或列表项只展示高价值信息：

- 渠道名称
- 配置状态
- 运行状态
- 一个高价值摘要

高价值摘要示例：

- 已连接
- 待扫码
- 需要补充凭证
- 默认投递对象已设置

### 配置方式原则

如果某个渠道存在引导式接入能力，应优先采用引导式接入，而不是让用户先看到一整页手填字段。

例如：

- 微信优先二维码登录流
- OAuth 类渠道优先授权流
- 手动 `.env` 字段作为高级兜底

### 微信特殊要求

微信接入必须支持内嵌二维码流程：

- 用户点击“连接微信”
- 页面内弹出二维码
- 展示扫码、确认、成功、过期等状态
- 不要求用户跳去浏览器完成主流程

外部打开页面只能作为备选，不应作为主路径。

### 默认隐藏的信息

- 大量原始 env 键值
- 技术字段说明
- 低价值状态解释
- 备份文件列表

这些内容可以收在“高级”或“更多”里。

### 二级动作

类似“备份并清空”“恢复备份”“查看 env 原文”属于次级动作，不应压过连接主流程。

---

## 5.4 技能页

技能页不是技能数据库浏览器，而是能力管理页。

### 主任务

- 查看当前有哪些技能可用
- 启用或禁用技能
- 安装新技能
- 删除本地技能

### 默认展示

每个技能条目默认只需要：

- 名称
- 一句话描述
- 来源
- 启用状态

### 可以二级展示的信息

- 标签
- 版本
- 所在位置
- 分类
- 平台限制

这些信息只有在筛选、排障或精细管理时才有价值，不应默认抢注意力。

### 交互要求

- 搜索必须明显
- 启用/禁用动作要直接
- 安装入口要短
- 详情可展开，但不要把整页做成信息卡墙

### 不应该出现的页面倾向

- 大量统计卡
- 路径和配置文件先行
- “能力总览”式包装

---

## 5.5 任务页

任务页服务的是“定时执行”，不是“系统监控”。

### 主任务

- 看有哪些任务
- 看任务何时执行
- 新建任务
- 手动运行 / 暂停 / 恢复 / 删除

### 默认展示

任务列表默认展示：

- 任务名称
- 调度描述
- 启用状态
- 下次执行时间
- 最近一次执行结果

### 创建任务表单

默认只要求用户填写：

- 名称
- 执行内容
- 调度方式
- 投递位置
- 附加技能（可选）

高级 cron 表达式或底层字段应作为折叠项，而不是默认入口。

### 调度器状态

调度器未运行时，需要明显提醒，但表达方式应克制：

- 一个状态横条或行内警告足够

不要把整页变成“Gateway 解释页”。

### 默认隐藏的信息

- 原始 jobs JSON
- 过多执行日志
- 大段错误解释
- 系统实现细节

查看执行详情应该发生在单个任务内部，而不是列表层。

---

## 5.6 设置页

设置页是偏好设置，不是知识中心，也不是系统总览。

### 主任务

- 让用户完成个人偏好和核心配置
- 在需要时进入高级设置

### 结构原则

设置页应使用桌面客户端偏好设置结构：

- 左栏：设置分类
- 中栏：当前分类下的对象或分段
- 右栏：当前对象的实际表单

当前对象与编辑内容必须同屏。

### 顶层分类建议

- 通用
- 助手
- 模型
- 语音与记忆
- 高级

### 设置页通用要求

- 不做概览卡墙
- 不做运行态总览页
- 不做解释性摘要区
- 不做“点击上面，表单出现在页面另一头”的布局

---

## 6. 模型设置页详细需求

模型是设置页里最重要的一项，但默认仍然必须简单。

### 6.1 页面定位

模型页只解决三件事：

1. 选定一个主模型
2. 确认连接可用
3. 在需要时调整高级模型能力

### 6.2 页面命名

一级入口使用：

- `模型`

不要默认使用：

- 模型与路由
- 路由矩阵
- 任务路由

### 6.3 内部分段

模型页只分三个分段：

- 基础
- 高级
- Provider

### 6.4 基础分段

这是普通用户默认看到的内容。

#### 目标

让用户以最短路径完成：

- 选择 Provider 类型
- 输入 URL / 登录凭证
- 自动获取模型
- 选择主模型
- 保存

#### 基础分段字段

- Provider 类型
- 接入方式
- Base URL
- API Key 或登录状态
- 自动探测模型
- 主模型选择器
- 测试连接
- 保存

#### 默认行为

1. 用户填写 URL 与 Key
2. 系统尝试获取 `/models`
3. 成功则填充模型列表
4. 默认推荐一个主模型
5. 用户直接保存

如果失败：

- 不阻塞流程
- 允许手动输入 model id
- 用一句短文案提示失败

#### 基础分段默认隐藏

- 压缩模型
- 辅助模型
- 子代理模型
- 回退策略
- 智能路由
- Provider catalog
- 模型目录元数据

### 6.5 高级分段

高级分段只服务少量高级用户，并且必须对齐 Hermes 真实能力：

- 压缩
- 辅助任务
- 子代理
- 回退策略
- 智能路由

不要再发明“任务路由”型抽象。

### 6.6 Provider 分段

Provider 分段的目标是把“自定义接口”产品化，而不是让用户理解配置树。

#### Provider Profile 至少包含

- 名称
- Provider 类型
- Base URL
- API Key
- 默认模型
- 已探测模型目录

#### 普通用户默认流程

1. 新建自定义接口
2. 输入 URL / Key
3. 自动探测模型
4. 选一个主模型
5. 保存

#### 高级用户才需要的能力

- 勾选要缓存的模型目录
- 编辑模型元数据
- 维护多个 Provider Profile

### 6.7 产品判断

普通用户只应该被要求配置一个主模型。

压缩、辅助、子代理、fallback、smart routing 这些能力都保留，但默认不应该构成心智门槛。

---

## 7. 文案与交互规范

## 7.1 文案规范

默认文案只做三件事：

- 告诉用户当前对象是什么
- 告诉用户当前状态是什么
- 告诉用户下一步能做什么

### 推荐风格

- 获取模型
- 测试连接
- 已连接
- 待扫码
- 未能自动获取模型，可手动填写
- 调度器未运行

### 禁止风格

- 当前重点
- 低价值信息收起
- 默认投递链路
- 先关注哪些……
- 这里展示的是……
- 你当前看到的是……

## 7.2 交互规范

- 选中态必须明确但克制
- hover 只做轻提示，不做戏剧化反馈
- 展开详情时尽量局部展开，不整页跳动
- 保存、测试、连接这类动作必须靠近对应对象
- 路由切换不能改变主题状态

---

## 8. 视觉方向

## 8.1 总体气质

界面应该是：

- 清爽
- 明亮
- 通透
- 克制
- 偏桌面客户端

而不是：

- 厚重
- 海报化
- 控制台化
- 工作台化
- 运维面板化

## 8.2 视觉元素

- 亮色优先
- 暖中性色为主
- 品牌色只作点缀
- 轻边框
- 轻阴影
- 小到中等圆角
- 列表与表单优先

### 推荐

- 单层主表面
- 清晰分栏
- 轻量 segmented control
- 抽屉、弹层、局部展开

### 避免

- 多层嵌套卡片
- 大面积强调色底板
- 顶部海报
- 页面内大段说明区
- 无意义装饰头像

---

## 9. 高保真设计交付要求

如果要交给设计模型出高保真，至少要覆盖下面这些关键画面：

1. 对话页默认态
2. 对话页包含 `thinking / tool use / 子代理 / 任务列表 / Markdown` 的复杂消息态
3. 渠道页默认态
4. 微信二维码接入弹层
5. 技能页默认态
6. 任务页默认态
7. 任务新建态
8. 设置页默认态
9. 模型设置页基础分段
10. 模型设置页高级分段
11. 模型设置页 Provider 分段

### 高保真必须遵守

- 不新增“总览”“工作台”“控制台”页面
- 不把低价值信息放到首屏
- 不出现自解释式文案
- 当前对象与操作要同屏
- 整体风格更接近桌面客户端偏好设置，而不是 SaaS 管理后台

---

## 10. 非目标

这次页面重置不追求下面这些方向：

- 做一个复杂的 AI workbench
- 做一个工程态监控中心
- 用更多信息来制造“功能很多”的感觉
- 在 UI 里讲 Hermes 内部实现
- 让普通用户理解全部模型层级

---

## 11. 结论

新的页面需求应该收敛到一句话：

> 聊天时就只让用户聊天，配置时就只让用户配置，只有在高级场景里才露出 Hermes 的复杂度。

因此，新的页面设计必须同时满足：

- Chat-first
- Simple-first
